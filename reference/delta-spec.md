# Delta-спецификация: приведение UI в соответствие с IA пространств
**Версия:** 1.0  
**Статус:** Готово к реализации  
**Базовые документы:** workspace.md v1.0, agent_editor.md, space-ia-spec.md v0.5

---

## Контекст

Данная спецификация описывает только изменения и новые элементы относительно существующих UI-спецификаций. Всё, что здесь не упомянуто, остаётся без изменений.

---

## Часть 1. Workspace — изменения

### 1.1 Удалить блок «Знания компании» из левой панели

**Текущее поведение:** левая панель содержит два раздела — «Исходные данные» и «Знания компании». Пользователь выбирает топики и файлы знаний через чекбоксы, они попадают в контекст чата.

**Причина изменения:** согласно IA, знания компании — компонент агента, а не пространства. Агент самостоятельно определяет, какие базы знаний ему нужны. Пользователь не управляет этим выбором во время сессии.

**Что нужно сделать:**
- Удалить секцию «Знания компании» из левой панели целиком (включая дерево топиков, чекбоксы, индикатор контекста).
- Удалить из индикатора контекста (нижняя часть левой панели) счётчик файлов знаний. Оставить только счётчик документов: «В контексте: N документов».
- Левая панель теперь содержит только один раздел — «Исходные данные» (документы IN).

---

### 1.2 Переименовать «Исходные данные» → «Документы»

**Причина:** IA использует термин «документы пространства» и «документы сессии» (IN/OUT). «Исходные данные» — неточный термин, не отражающий IA-модель.

**Что нужно сделать:**
- Переименовать заголовок секции с «Исходные данные» на «Документы».
- Обновить все пустые состояния и подсказки, где упоминаются «исходные данные».
- Кнопку «Сохранить в Исходные данные» в правой панели (вкладка «Артефакт») переименовать в «Сохранить в документы пространства».

---

### 1.3 Обновить точки входа в сессию

**Текущее поведение:** в правой панели (вкладка «Агенты и навыки») есть режим «Свободный режим» и список агентов. Пользователь либо остаётся в свободном режиме, либо выбирает агента.

**Причина изменения:** IA определяет три явные точки входа в сессию: выбор агента, вызов навыка через `/команду`, свободная задача. По умолчанию используется универсальный агент.

**Что нужно сделать:**

Переработать секцию «Режим работы» в правой панели:

```
┌─────────────────────────────────────┐
│ НОВАЯ СЕССИЯ                        │
├─────────────────────────────────────┤
│ ○  Универсальный агент  [по умолч.] │
│    Начните с задачи или /навыка     │
├─────────────────────────────────────┤
│ ●  Преднастроенные агенты           │
│    ┌─────────────────────────────┐  │
│    │ [Иконка] Юридический       │  │
│    │ Анализ договоров            │  │
│    │                  [Выбрать]  │  │
│    └─────────────────────────────┘  │
│    ┌─────────────────────────────┐  │
│    │ [Иконка] Ассистент встреч  │  │
│    │ Протоколы и задачи          │  │
│    │                  [Выбрать]  │  │
│    └─────────────────────────────┘  │
└─────────────────────────────────────┘
```

- Пункт «Свободный режим» → «Универсальный агент» с пометкой «по умолчанию».
- Агенты в списке — только преднастроенные. Активный агент подсвечен, остальные кликабельны.
- Выбор агента фиксирует его для текущей сессии. После фиксации список сворачивается, отображается только активный агент с кнопкой «Сменить» (которая сбрасывает на универсальный и начинает новую сессию, т.к. смена агента внутри сессии невозможна).

---

### 1.4 Обновить пространство имён слеш-команд

**Текущее поведение:** ввод «/» показывает единый список навыков.

**Причина изменения:** IA определяет два уровня навыков с разным синтаксисом — `/название` для компании, `./название` для пространства.

**Что нужно сделать:**
- Выпадающий список при вводе `/` делится на две секции с заголовками: **Навыки компании** и **Навыки пространства**.
- В каждой строке списка отображается: иконка, название, краткое описание.
- При выборе навыка из секции «Навыки компании» в поле ввода подставляется `/название`.
- При выборе навыка из секции «Навыки пространства» подставляется `./название`.
- Фильтрация по символам после `/` работает по обеим секциям одновременно; при фильтрации заголовки секций остаются видимыми.
- Если пользователь вручную вводит `./` — список сразу показывает только навыки пространства.

### 1.5 Новое: состояние сессии «Ожидание»

**Текущее поведение:** состояние «Ожидание» в workspace.md не описано. Есть только блокировка ввода во время исполнения.

**Причина добавления:** IA определяет статус «Ожидание» как отдельный статус сессии — агент дошёл до шага, требующего действия человека, либо обнаружил отсутствие учётных данных для инструмента.

**Что нужно сделать:**

#### 1.5.1 Карточка «Ожидание» в чате

Когда агент переходит в статус «Ожидание», в конце истории чата появляется системная карточка:

```
┌─────────────────────────────────────────────────┐
│ ⏸  Агент ожидает вашего действия                │
│                                                 │
│  [Причина: текст от агента — что нужно сделать] │
│                                                 │
│           [Продолжить]    [Завершить сессию]    │
└─────────────────────────────────────────────────┘
```

- Кнопка «Продолжить» разблокирует поле ввода и переводит сессию обратно в «Активна».
- Кнопка «Завершить сессию» переводит сессию в статус «Завершена».

#### 1.5.2 Специальный вариант: отсутствие учётных данных инструмента

Если причина перехода в «Ожидание» — отсутствие учётных данных для MCP-инструмента:

```
┌─────────────────────────────────────────────────┐
│ ⚠️  Требуется настройка инструмента             │
│                                                 │
│  Для продолжения работы необходимо подключить:  │
│  • [Название инструмента]                       │
│                                                 │
│  [Настроить инструмент]    [Отмена]             │
└─────────────────────────────────────────────────┘
```

- Кнопка «Настроить инструмент» открывает панель настройки учётных данных (в рамках пользовательского профиля или inline-форму — на усмотрение реализации).
- После успешной настройки сессия автоматически продолжается без дополнительного действия пользователя.

#### 1.5.3 Поле ввода в статусе «Ожидание»

- Поле ввода разблокировано (в отличие от статуса «Активна» во время исполнения).
- Плейсхолдер: «Ответьте агенту или нажмите Продолжить».

### 1.6 Новое: принятие триггерных сессий

**Текущее поведение:** не описано. Workspace предполагает что сессию всегда создаёт пользователь.

**Причина добавления:** IA определяет, что триггерные сессии (по расписанию или событию) стартуют без владельца и переходят в «Ожидание», пока участник не возьмёт сессию на себя.

**Что нужно сделать:**

#### 1.6.1 Список входящих сессий

В рабочем пространстве (на уровне навигации по сессиям, выше отдельной сессии) добавить раздел «Входящие» или бейдж-счётчик над списком сессий, показывающий количество сессий в статусе «Ожидание» без владельца.

Каждая такая сессия в списке отображается с бейджем **«Ожидает»** и кнопкой **«Взять»**.

#### 1.6.2 Карточка принятия сессии

При клике на «Взять» — модальное окно подтверждения:

```
┌─────────────────────────────────────────────────┐
│ Взять сессию                                    │
│                                                 │
│  Агент: [Название агента]                       │
│  Запущена: [дата и время]                       │
│  Триггер: [описание события]                    │
│                                                 │
│  После принятия вы станете владельцем сессии.   │
│  Другие участники смогут только читать её.      │
│                                                 │
│              [Отмена]    [Взять сессию]         │
└─────────────────────────────────────────────────┘
```

После принятия пользователь оказывается внутри сессии, которая уже содержит историю работы агента.

#### 1.6.3 Сессии в статусе «Ожидание» у других участников (режим «Общие сессии»)

В списке сессий пространства участники видят чужие сессии в статусе «Ожидание». У чужой сессии в «Ожидание» доступна кнопка «Взять» (переназначение). Сессии в статусе «Активна» доступны только для чтения, кнопки «Взять» нет.

### 1.7 Обновить вкладку «Артефакт» — кнопка сохранения в знания

**Текущее поведение:** кнопка «Сохранить в Знания компании» открывает выбор топика.

**Причина изменения:** в IA знания организованы как специализированные базы (юридическая, финансовая и т.д.), не как топики.

**Что нужно сделать:**
- При нажатии «Сохранить в базу знаний» открывать список доступных баз знаний компании (не дерево топиков).
- Если у пользователя нет прав на запись ни в одну базу — кнопка disabled с тултипом «Нет прав на редактирование баз знаний».
- После сохранения показывать inline-подтверждение: «Сохранено в [Название базы]».

## Часть 2. Agent Editor — изменения

### 2.1 Добавить блок KNOWLEDGE

**Текущее состояние:** блока нет.

**Причина добавления:** по IA, знания компании — компонент агента. Агент ссылается на одну или несколько баз знаний в своей конфигурации.

**Расположение на canvas:** справа, между блоками TOOLBOX и SKILLS (или отдельной строкой — см. обновлённую схему ниже).

**Обновлённая схема расположения блоков:**

```
[SCHEDULE]      [AGENT]           [TOOLBOX]
                [INSTRUCTIONS]
[CHANNELS]                        [KNOWLEDGE]

                                  [SUB-AGENTS]

                                  [SKILLS]
```

**Соединительная линия:** KNOWLEDGE → AGENT, цвет — зелёный (пунктир), аналогично SKILLS.

**Содержимое блока:**

```
┌────────────────────────────────────┐
│ KNOWLEDGE              [+ Connect] │
├────────────────────────────────────┤
│  Нет подключённых баз знаний       │  ← пустое состояние
└────────────────────────────────────┘
```

Заполненное состояние:

```
┌────────────────────────────────────┐
│ KNOWLEDGE              [+ Connect] │
├────────────────────────────────────┤
│  [🗂] Юридическая база        [🗑] │
│  [🗂] Нормативные документы   [🗑] │
└────────────────────────────────────┘
```

**Добавление:** кнопка `+ Connect` → попап в режиме каталога. Каталог показывает список баз знаний компании, к которым у пользователя есть доступ на чтение. Каждая позиция: иконка, название базы, краткое описание. Выбор — немедленное добавление и закрытие попапа. Одна база может быть подключена только один раз (уже подключённые — disabled в каталоге).

**Маппинг попапа:**

| Блок | Действие | Режим попапа |
|---|---|---|
| Knowledge | + Connect | Каталог (базы знаний компании) |

---

### 2.2 Изменить поведение блока SKILLS — создание навыка

**Текущее поведение:** кнопка `Create` в блоке SKILLS открывает текстовый редактор и создаёт навык, привязанный к агенту.

**Причина изменения:** по IA, навыки хранятся в пространстве или на уровне компании — не внутри агента. Агент только ссылается на навык. Создание навыка должно сохранять его в пространство, а не в конфигурацию агента.

**Что нужно сделать:**
- Поведение кнопки `Create` остаётся прежним (открывает попап с полем Name + markdown-редактор).
- Добавить в попап уточнение: «Навык будет сохранён в навыки пространства и станет доступен другим агентам и пользователям».
- После сохранения навык появляется и в блоке SKILLS агента (как ссылка), и в списке навыков пространства.
- Кнопка `+ Add` в блоке SKILLS при открытии каталога показывает два раздела: **Навыки пространства** и **Навыки компании** — с соответствующими префиксами (`./` и `/`) рядом с названием.

---

### 2.3 Изменить блок TOOLBOX — модель подключения MCP

**Текущее поведение:** кнопка `MCP` в блоке TOOLBOX открывает форму подключения нового MCP-сервера (URL, auth) непосредственно из редактора агента.

**Причина изменения:** по IA, MCP-инструменты компании управляются администратором компании в глобальном разделе. Агент только ссылается на уже подключённые инструменты. Создание нового MCP-сервера из редактора агента не соответствует модели прав.

**Что нужно сделать:**
- Кнопку `MCP` убрать из заголовка блока TOOLBOX для пользователей без роли «Администратор компании».
- Для администраторов компании — кнопка `MCP` остаётся, но при нажатии открывает глобальный раздел управления инструментами (переход или встроенный попап).
- Кнопка `+ Add` в блоке TOOLBOX открывает каталог только тех инструментов, которые уже подключены на уровне компании. Если инструментов нет — пустое состояние: «Инструменты не настроены — обратитесь к администратору».
- Бейдж `API Key Missing` на инструменте переименовать в **«Требует авторизации»** — он означает, что пользователь не настроил личные учётные данные для этого инструмента. Клик по бейджу открывает форму авторизации пользователя (не редактирование MCP-сервера).

---

### 2.4 Обновить блок SKILLS — отображение пространства имён

**Текущее поведение:** навыки в блоке отображаются просто по названию.

**Что нужно сделать:**
- Рядом с каждым навыком в блоке показывать его уровень в виде бейджа: `компания` или `пространство`.
- В каталоге выбора навыков (`+ Add`) — навыки сгруппированы в два раздела: **Навыки компании** (с префиксом `/`) и **Навыки пространства** (с префиксом `./`).

---

### 2.5 Уточнить блок CHANNELS в контексте IA

**Текущее состояние:** блок CHANNELS описывает каналы коммуникации (Messaging, Web, Email) как отдельную сущность.

**Соответствие IA:** в IA внешние триггеры — это компонент агента, включая расписание и события. Email-канал в этой модели является триггером: входящее письмо → запуск сессии.

**Что нужно сделать:**
- Переименовать блок `CHANNELS` → `TRIGGERS`.
- Объединить блоки `SCHEDULE` и `CHANNELS` (новое название `TRIGGERS`) в один блок, поскольку оба описывают условия запуска сессии.
- Обновлённый блок `TRIGGERS`:

```
┌────────────────────────────────────┐
│ TRIGGERS               [+ Add]     │
├────────────────────────────────────┤
│  Нет настроенных триггеров         │  ← пустое состояние
└────────────────────────────────────┘
```

- Кнопка `+ Add` → попап в режиме каталога. Категории: **Расписание** (Cron / Interval / One-time), **Событие** (Webhook, Email, очередь).
- Для триггеров типа «Email» — сохраняется логика идентификации (бывший `⚠️ Set identity`), но теперь как шаг настройки конкретного триггера, а не блока целиком.
- Соединительная линия TRIGGERS → AGENT: цвет янтарный, пунктир — сохраняется.

**Обновлённая схема расположения блоков:**

```
[TRIGGERS]      [AGENT]           [TOOLBOX]
                [INSTRUCTIONS]
                                  [KNOWLEDGE]

                                  [SUB-AGENTS]

                                  [SKILLS]
```

---

### 2.6 Блок SUB-AGENTS — статус

**Текущее состояние:** блок описан в спецификации редактора агентов.

**Соответствие IA:** понятие «под-агент» в текущей версии IA не определено. Блок не противоречит архитектуре, но выходит за её рамки.

**Решение:** блок SUB-AGENTS оставить без изменений. В рамках данного delta считается вне скоупа — требует отдельного решения об архитектуре под-агентов.

## Часть 4. Вне скоупа данного delta

Следующие темы требуют отдельных решений и не входят в данную спецификацию:

- Архитектура под-агентов (блок SUB-AGENTS)
- Мобильная версия workspace
